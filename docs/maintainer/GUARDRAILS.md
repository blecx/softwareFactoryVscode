# Maintainer guardrail catalog

This page helps maintainers find the current enforcement surfaces across the repository.
It is an index/reference, not a competing normative authority.
Accepted ADRs, repository instructions, templates, workflow modules, and configuration files remain the real source of truth.

## How to use this catalog

- Start with the guardrail family that matches your question.
- If you already know the workflow name and need the fastest operational map,
  jump to [`AGENT-ENFORCEMENT-MAP.md`](AGENT-ENFORCEMENT-MAP.md).
- Follow the linked source files before changing behavior, wording, or workflow expectations.
- If two surfaces appear to overlap, prefer accepted ADRs for architecture authority, templates for issue/PR structure, workflow modules for issue execution, and concrete config/scripts for approval behavior.

## Fast lookup

- **Architecture or documentation authority drift:** start with `ADR-013` and the accepted ADRs in `docs/architecture/`.
- **Future-project wiki bootstrap or ownership split:** start with `docs/maintainer/HOST-WIKI-TRUTH-CONTRACT.md`, `docs/WIKI-MAP.md`, and `manifests/wiki-projection-manifest.json`.
- **Live wiki sync/publishing after host truth exists:** start with `docs/maintainer/WIKI-PUBLISHING.md`, `docs/WIKI-MAP.md`, and `manifests/wiki-projection-manifest.json`.
- **Need the workflow-specific control chain quickly:** start with `docs/maintainer/AGENT-ENFORCEMENT-MAP.md`.
- **Prompt workflow entrypoints:** start with `docs/maintainer/PROMPT-WORKFLOWS.md` and `.github/prompts/*`.
- **Issue/PR execution behavior:** start with `docs/WORK-ISSUE-WORKFLOW.md` and the relevant `.copilot/skills/*` module.
- **Queue/interruption re-anchoring or stale editor paths:** start with `.github/copilot-instructions.md`, `docs/WORK-ISSUE-WORKFLOW.md`, and `.tmp/github-issue-queue-state.md` before trusting any `.tmp/queue-worktrees/*` editor path.
- **Issue or PR structure requirements:** start with `.github/ISSUE_TEMPLATE/*`, `.github/pull_request_template.md`, and `scripts/validate-pr-template.sh`.
- **Local-vs-remote validation expectations:** start with `.github/workflows/ci.yml`, `ADR-006`, and `scripts/local_ci_parity.py`.
- **Shared validation runner/report contract:** start with `docs/maintainer/VALIDATION-RUNNER-CONTRACT.md`, `factory_runtime/agents/validation_runner.py`, `factory_runtime/agents/validation_plan_resolver.py`, and `configs/validation_policy.yml` before altering bundle execution semantics or structured result fields.
- **Writing or reviewing CI-sensitive tests/wrappers:** start with `tests/README.md` and keep blocking failures fast-fail/actionable before leaning on watchdogs.
- **Formatter fidelity for generated Python files:** start with `.github/copilot-instructions.md`, `tests/README.md`, and the repo-owned Python write helpers before trusting hand-formatted output.
- **Approval profiles or workspace tool policy:** start with `docs/maintainer/APPROVAL-PROFILES.md`, `.vscode/tasks.json`, `scripts/setup-low-approval.sh`, `scripts/setup-vscode-agent-settings.py`, and `configs/bash_gateway_policy.default.yml`.

## Scope note

This catalog deliberately does **not** restate every workflow rule or flatten all enforcement surfaces into one pseudo-spec.
Use it to find the right source file quickly, then edit that source directly.
