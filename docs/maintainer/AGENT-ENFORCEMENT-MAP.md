# Agent enforcement map

This page is the workflow-specific companion to [`GUARDRAILS.md`](GUARDRAILS.md).
Use it when you already know the major agent or workflow entrypoint and need the
fastest truthful answer to “which surface actually enforces this one?”

It is an index/reference, not a competing normative authority.
Accepted ADRs, repository instructions, workflow skills, prompts, templates,
and validation scripts remain the real source of truth.

## How to read this map

- Start with the workflow or agent entrypoint being invoked.
- Follow the wrapper first, then the canonical skill, then any prompt,
  template, checkpoint, or ADR listed for that boundary.
- If a row routes back to `resolve-issue` → `pr-merge`, it is an orchestration
  wrapper, not a second implementation or merge workflow.

## Shared cross-workflow enforcement surfaces

- [`ADR-013`](../architecture/ADR-013-Architecture-Authority-and-Plan-Separation.md)
  defines why this page and [`GUARDRAILS.md`](GUARDRAILS.md) stay as indexes
  instead of pretending to replace the accepted ADRs or workflow skills.
- [`../../.github/copilot-instructions.md`](../../.github/copilot-instructions.md)
  is the repository-wide policy surface for tool routing, checkpoint
  discipline, execution surfaces, release guardrails, and ADR awareness.
- [`../WORK-ISSUE-WORKFLOW.md`](../WORK-ISSUE-WORKFLOW.md) is the canonical
  cross-agent sequence map tying `create-issue`, `resolve-issue`,
  `pr-merge`, queue wrappers, and interruption recovery into one supported
  issue → PR → merge contract.
- [`ADR-005`](../architecture/ADR-005-Strong-Templating-Enforcement.md) and
  [`ADR-006`](../architecture/ADR-006-Local-CI-Parity-Prechecks.md) are the
  two workflow-level architecture guardrails that appear repeatedly because
  template discipline and local CI parity are mandatory across the PR path.
